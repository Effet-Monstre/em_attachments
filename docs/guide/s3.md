# S3 Backend

`EmAttachments.Backends.S3` stores files on Amazon S3 (or any S3-compatible service) using AWS Signature v4. No ExAws dependency — the backend signs requests directly via `req`.

---

## Basic configuration

```elixir
config :em_attachments, :config,
  secret_key: System.get_env("EM_SECRET_KEY"),
  store: {EmAttachments.Backends.S3,
    bucket: "my-bucket",
    prefix: "uploads",
    region: "us-east-1",
    access_key_id: {:env, "AWS_ACCESS_KEY_ID"},
    secret_access_key: {:env, "AWS_SECRET_ACCESS_KEY"},
    acl: :public_read},
  cache: [prefix: "cache"]   # inherits store's backend and credentials
```

| Option | Default | Description |
|---|---|---|
| `bucket` | — | S3 bucket name (required) |
| `prefix` | `"uploads"` | Key prefix for all stored objects |
| `region` | `AWS_REGION` env / `"us-east-1"` | AWS region |
| `access_key_id` | `AWS_ACCESS_KEY_ID` env | AWS access key ID |
| `secret_access_key` | `AWS_SECRET_ACCESS_KEY` env | AWS secret access key |
| `acl` | `:private` | Canned ACL applied on upload |
| `url_expires_in` | `3600` | Presigned URL TTL in seconds |
| `policy` | — | Set to `:cache` to enable the cache policy (see below) |
| `cache_ttl` | `1800` | Seconds before a cache-policy file is auto-deleted |

### ACL options

| Value | S3 header |
|---|---|
| `:private` | `x-amz-acl: private` |
| `:public_read` | `x-amz-acl: public-read` |
| `:authenticated_read` | `x-amz-acl: authenticated-read` |

When `acl: :public_read`, `url/2` returns the plain object URL. Otherwise it returns a presigned URL valid for `url_expires_in` seconds.

### Credentials via environment variables

Any option value can be an `{:env, "VAR"}` tuple — it is resolved at runtime via `System.get_env/1`:

```elixir
{EmAttachments.Backends.S3,
  bucket: {:env, "S3_BUCKET"},
  access_key_id: {:env, "AWS_ACCESS_KEY_ID"},
  secret_access_key: {:env, "AWS_SECRET_ACCESS_KEY"},
  region: {:env, "AWS_REGION", "us-east-1"}}  # third element is a fallback
```

## Server-side copy on promotion

When both the cache and store backends use the same S3 bucket, promotion is a server-side `CopyObject` — the file is never downloaded to your server. When using the cache policy (see below), the copy is skipped entirely because the file is already at the store location.

---

## Presigned uploads

For direct browser-to-S3 uploads, generate presigned POST credentials:

```elixir
{:ok, %{url: url, fields: fields}} = MyUploader.presign_upload()
```

Render `url` and `fields` as a multipart HTML form. The browser uploads directly to S3. After the upload completes, call `MyUploader.upload/1` with the returned object key to run the plugin pipeline.

## Cache policy

The cache policy is an optimisation for applications where the cache and store backends point to the same S3 bucket and key prefix. Instead of uploading the file twice (once to cache, once to store on promotion), the file is written directly to its **permanent store location** during the cache phase. A lightweight sentinel object tracks that the file is pending promotion, and a timer deletes it automatically if promotion never happens.

### How objects are laid out

| Phase | S3 key | Content |
|---|---|---|
| After `upload/1` | `{prefix}/{id}` | The actual file at its final location |
| | `{prefix}/cache/{id}` | Empty sentinel object |
| After `promote/1` | `{prefix}/{id}` | File unchanged — already in place |
| | `{prefix}/cache/{id}` | Deleted |
| After TTL expiry | `{prefix}/{id}` | Deleted |
| | `{prefix}/cache/{id}` | Deleted |

### Setup

Configure the cache backend with `policy: :cache` and the **same prefix** as the store:

```elixir
config :em_attachments, :config,
  store: {EmAttachments.Backends.S3,
    bucket: "my-bucket",
    prefix: "uploads",
    region: "us-east-1",
    access_key_id: {:env, "AWS_ACCESS_KEY_ID"},
    secret_access_key: {:env, "AWS_SECRET_ACCESS_KEY"},
    acl: :private},
  cache: [
    policy: :cache,
    prefix: "uploads",   # must match store prefix
    cache_ttl: 1800      # seconds until auto-delete if not promoted (default 1800)
  ]
```

The cache backend inherits all credentials and options from the store backend. Only the keys that differ need to be provided.

### CacheRegistry

The cache policy requires `EmAttachments.Backends.S3.CacheRegistry` to be running. Add it to your application's supervision tree:

```elixir
defmodule MyApp.Application do
  use Application

  def start(_type, _args) do
    children = [
      MyApp.Repo,
      EmAttachments.Backends.S3.CacheRegistry,   # add this
      MyAppWeb.Endpoint
    ]

    Supervisor.start_link(children, strategy: :one_for_one)
  end
end
```

When started with no options, the registry **auto-discovers** any S3 backends with `policy: :cache` from `config :em_attachments, :config`. For the common single-backend case this is all you need.

#### Explicit backend list

If you manage credentials differently or have multiple S3 backends, pass the backends explicitly:

```elixir
{EmAttachments.Backends.S3.CacheRegistry,
  backends: [
    [
      bucket: "my-bucket",
      prefix: "uploads",
      region: "us-east-1",
      access_key_id: System.get_env("AWS_ACCESS_KEY_ID"),
      secret_access_key: System.get_env("AWS_SECRET_ACCESS_KEY"),
      policy: :cache,
      cache_ttl: 1800
    ]
  ],
  cleanup_interval: :timer.hours(12)}
```

| Option | Default | Description |
|---|---|---|
| `backends` | auto-discovered | List of backend opts keyword lists to scan |
| `cleanup_interval` | 24 hours | Milliseconds between periodic cleanup scans |

### Promotion flow

When `promote/1` is called, the pipeline calls `store_mod.put(id, cache_backend_file, store_opts)`. The backend detects that this is a same-bucket, same-key, cache-policy copy and short-circuits:

1. **No HTTP request is made** — the file is already at its final location.
2. The CacheRegistry timer is cancelled immediately.
3. All sentinel objects are deleted in a single S3 Delete Objects batch call.

This means promotion is essentially free for same-bucket setups: one batch API call instead of a full upload.

### Automatic expiry

When a file is uploaded but never promoted — for example when a user abandons a form mid-session — the CacheRegistry fires a timer after `cache_ttl` seconds and deletes both the file and the sentinel from S3.

### Restart and crash recovery

Timer state is held in memory by CacheRegistry. If the process crashes or the application restarts, in-flight timers are lost. At startup (and periodically thereafter according to `cleanup_interval`), the registry runs a **cleanup scan**:

1. Lists all objects under `{prefix}/cache/` — only the small set of pending sentinels.
2. Compares each sentinel's `LastModified` timestamp against `now - cache_ttl`.
3. Deletes any files whose sentinels are older than the TTL.

Because the sentinel prefix is narrow, this scan is fast regardless of how many permanent files exist in the bucket.

### DB failure safety

Since S3 operations are not transactional, a file may be permanently stored even if the database write fails. This is intentional and safe:

- The file sits at its store location and is indistinguishable from any other stored file.
- The sentinel is deleted during promotion, so the cleanup scan will not touch it.
- If the user resubmits the form, the serialised cache JSON is re-promoted as a no-op — the file is already in place, the timer cancel is a no-op, and the DB write succeeds.

### Per-uploader override

Apply the cache policy to a specific uploader rather than globally:

```elixir
defmodule MyApp.AvatarUploader do
  use EmAttachments.Uploader,
    cache: [policy: :cache, prefix: "uploads", cache_ttl: 3600],
    store: {nil, [prefix: "uploads"]}  # nil inherits the global store backend
end
```

`{nil, [prefix: "uploads"]}` uses the globally configured store backend but overrides the prefix.

## Bulk operations

The S3 backend implements the optional `bulk_delete/2` and `bulk_put/2` callbacks, which the library uses automatically when available.

### `bulk_delete`

Uses the S3 Delete Objects API to remove up to 1000 keys per request. The library calls this when destroying a file that has many derivatives, or when the CacheRegistry expires a batch of sentinels:

```elixir
# Called automatically — no user code required.
# Exposed if you need it directly:
S3.bulk_delete(["id1", "id2", "id3"], opts)
```

### `bulk_put`

Runs multiple PUT or CopyObject operations in parallel. Promotions within the same bucket are grouped so their sentinel deletes are batched into a single Delete Objects call. All other puts run concurrently via `Task.async_stream`.

The single-file `put/3` delegates to `bulk_put/2` internally, so the same parallel pipeline is used for every upload.
